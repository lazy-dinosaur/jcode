## 2. 구현 내용 (M19)

### 코드 변경

- `src/config.rs`
  - `OnceLock<Config>` → `OnceLock<Mutex<ConfigCache>>` 로 교체.
  - `config()` 가 매 호출에서 `stat()` → mtime 비교 (500ms debounce).
    변경 감지 시 새 `Config` 파싱하여 `Box::leak` (그래야 기존 `&'static Config`
    public signature 유지). 이전 leak 은 그대로 살아있음 (안전).
  - `force_reload_config()` (set_default_model 등 명시 reload 진입점).
  - `reset_config_cache_for_tests()` `#[cfg(test)]` 전용.
  - parse error 시 last-good 유지 (warn 만 로깅).
- `src/config/config_file.rs`
  - `Config::try_load() -> anyhow::Result<Self>` 추가 (parse 실패를
    호출자가 구분할 수 있도록).
  - `set_default_model` 안의 옛 `CONFIG.get_or_init` 우회 로직 제거 →
    `force_reload_config()` 호출 + 주석으로 “M19 가 restart 불필요”명시.
- `src/config_tests.rs`
  - 5개 테스트 추가 (모두 `crate::storage::lock_test_env()` 로 직렬화):
    1. `test_m19_config_initial_load_after_reset`
    2. `test_m19_config_reloads_when_mtime_changes`
    3. `test_m19_config_keeps_last_good_on_invalid_toml`
    4. `test_m19_config_recovers_after_invalid_then_valid_write`
    5. `test_m19_force_reload_is_noop_when_unchanged`
  - 토글 필드는 `display.pin_images` (default `true`) 사용 —
    `show_token_count` 는 존재하지 않음.

### 검증

```
cd /home/lazydino/dev/jcode
git checkout patch/config-hot-reload
cargo test --lib config::tests::test_m19 -- --test-threads=1 --nocapture
# → running 5 tests; test result: ok. 5 passed; 0 failed; ...
```

(주의: 첫 컴파일 ~3분 30초. M20 timeout 회피용으로
`nohup cargo test ... > /tmp/m19-test.log 2>&1 & disown` 권장.)

### 디자인 메모

- **leak 전략 정당화**: 단위 시간당 config 변경 횟수가 매우 작고
  Config struct 크기가 작아서 leak 누적 비용 무시 가능. 대신
  `&'static Config` API 가 깨지지 않는다는 큰 이점.
- **debounce 500ms**: editor save 시 같은 mtime 으로 여러 번 stat 호출
  되는 burst 를 한 번의 reload 로 수렴.
- **last-good fallback**: 사용자가 toml 편집 중간에 파싱 깨진 상태로
  저장하더라도 process 가 default 로 떨어지지 않고, 다음 valid
  edit 까지 직전 상태 유지.

## 3. cherry-pick conflict 진행 (미완)

`patch/config-hot-reload` 의 두 커밋을 `deploy/m9-m10` 으로 cherry-pick 시도:

```
git checkout deploy/m9-m10
git cherry-pick 93282b44 bdf63f5c   # 첫째 OK, 둘째에서 conflict
```

`src/config_tests.rs` 의 `use super::{...}` import 줄과 두 군데 본문
(M9 test 영역과 M19 test 영역) 에서 충돌.

세션 중간에 단순 `sed -i '/^<<<<<<< HEAD$/d; /^=======$/d; /^>>>>>>>` 로
마커만 지웠는데 — closing brace 가 마커 안에 묻혀서 같이 사라지지 않은
듯하지만 실제로 최종 파일에서 `} - unclosed delimiter` 가 나옴
(M9 test fn 의 `assert_eq!(...)` 영역과 fn 닫기 `}` 가 섞임).

→ **abort 후 미해결 상태로 deploy/m9-m10 깨끗**. (현재 git 상태:
`git status` clean, deploy 가 `f9883ee2` 그대로.)

### 다음 세션 권장 절차

`patch/config-hot-reload` 를 deploy 에 통합하는 더 깔끔한 방법:

```
git checkout patch/config-hot-reload
git rebase deploy/m9-m10
# config_tests.rs 충돌 한 번만 풀고:
#  - HEAD (deploy) 쪽: M9 test, hooks_for_working_dir_dedupes_*, ...
#  - incoming (M19): test_m19_*, m19_write_config helper, M19 import 변경
# 두 영역 모두 keep, brace 카운트 맞추기.
git add src/config_tests.rs
git rebase --continue

# 그리고 deploy 에 fast-forward 가 안 되면 별도 통합 브랜치 만들기:
git checkout deploy/m9-m10
git merge --ff-only patch/config-hot-reload   # 또는 새 deploy/m9-m10-m19
```

또는 통합 브랜치 새로 파는 쪽이 안전 (deploy 이력 보존):

```
git checkout -b deploy/m9-m10-m19 deploy/m9-m10
git cherry-pick 93282b44      # fix only
# config_tests.rs 만 수동 merge (M9 + M19 test 둘 다 keep)
git cherry-pick bdf63f5c
```

**핵심 주의**: `src/config_tests.rs` 의 `use super::{...}` 줄이
M9 test 가 추가한 `AgentRouteConfig`, `HookCommandConfig`, `OsStr`,
`OsString`, `EnvVarGuard` 등을 이미 가져왔는데 M19 commit 은 그걸
모르고 import 를 더 좁힌 상태이므로, **양쪽 import 를 union** 해야 한다.

## 4. M20 — jcode bash tool 의 2분 hard timeout (신규)

### 증상

- 메인 세션의 `Bash` tool 호출이 2분 도달 시 강제 background 이동.
- cargo test (~3-4분), cargo build (~5분+), 긴 sleep, 긴 stress 테스트
  모두 timeout 메시지 (`Tool 'bash' was moved to background by the user`)
  로 끝나며 사용자에게 “실패한 것처럼” 보임.
- nohup + disown 으로 process 자체는 살아남지만, **호출 메시지 자체가
  끊겨** 결과 수집이 폴링 기반이 됨.

### 위치

- `src/tool/bash.rs:23` — `const DEFAULT_TIMEOUT_MS: u64 = 120000;` 하드코딩.
- 이 값은 모든 `bash`/`shell` tool 호출에 일괄 적용.

### 영향 받는 작업

- M19 `cargo test` (~3분 30초 컴파일).
- M11 통합 빌드.
- 모든 `cargo build --release`.
- 일반적인 long-running 검증.

### 해결 방향 (옵션)

- **A. config 로 노출**: `[tool.bash] default_timeout_ms = 600000` 같은
  키를 `Config` 에 추가, `bash.rs` 가 읽도록. (~30 LOC)
- **B. tool 호출 인자에 노출**: `Bash` tool 의 input schema 에
  `timeout_ms` field 추가, 모델이 명시적으로 길게 줄 수 있게. (~50 LOC)
- **C. heuristic**: cargo/build 명령 자동 감지 후 timeout 상향. (취약)

**권장: A + B** 동시. config default 는 10분, 호출 시 override 가능.

### 검증 시나리오

- A 만: config 에 60초로 줄여서 짧은 sleep 도 timeout 되는지 확인.
- B 만: `timeout_ms=10000` 명시 후 `sleep 30` 호출 → 정확히 10초에 종료.

### 우선순위

M19 통합 직후. M11 작업 시작 전. 안 그러면 모든 빌드 검증이 폴링 곡예가 됨.

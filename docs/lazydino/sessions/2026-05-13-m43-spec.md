## 0. 한 줄 요약

`src/provider/anthropic.rs::AnthropicProvider::format_tools(tools, is_oauth=true)`
가 **하드코딩된 10개 tool 만** 광고하고 `tools` 인자를 완전히 무시한다.
이 때문에 jcode-only tool (`bg`, `bash_output`, `bash_kill`, `swarm`,
`todo*` 등) 은 OAuth 경로 LLM 한테 **절대 보이지 않는다**. 메인
세션, subagent 둘 다 동일하게 적용된다.

## 1. 배경 & 사용자 영향

### 1.1 사용자 증상 (라이브 관찰)

- 메인 세션이 system reminder `bg action="output" task_id=...` 받음.
  subagent 에 위임 → "내 도구 목록에 bg 없음" 실패. 결국 메인이
  직접 호출해야 한다는 걸 깨닫는 데 라운드 낭비.
- 메인 세션도 swarm 도구가 안 보임 → swarm worker spawn 시도 불가.

### 1.2 영향 범위

OAuth (claude-* via subscription) 로 실행되는 **모든 세션** — 메인,
subagent, swarm worker (있다면). API key path (Anthropic API key,
OpenAI 등) 는 영향 없음 (해당 path 는 `is_oauth=false` 분기).

### 1.3 누락되는 tool 목록 (확인된 것)

`src/tool/mod.rs:160-230` 의 base tool 등록 vs `format_tools` 의
10개 광고 비교:

**광고됨 (10):** `Agent` (=subagent), `Bash`, `Edit`, `Glob`, `Grep`,
`Read`, `ScheduleWakeup` (=schedule), `Skill` (=skill_manage),
`ToolSearch` (=codesearch), `Write`.

**누락 (M43 영향):**
- `bg` — background task 조회/관리 (핵심)
- `bash_output` — bash background output 폴링
- `bash_kill` — bash background 종료
- `swarm` (=communicate) — multi-session 코디네이션 (핵심)
- `todo` / `todowrite` / `todoread` — task tracking
- `memory` — long-term memory
- `selfdev` — canary 모드 (canary session 만)
- `mcp_*` — 모든 MCP tool
- 그 외 base 에 등록된 모든 jcode-내부 tool

(완전 목록은 `format_tools` 와 `register_base_tools` 의 diff 로
확정. 명세 §4.1 참고.)

## 3. Fix 설계

### 3.1 핵심 의사결정

OAuth path 에서 `tools` 인자를 **사용**해야 한다. 단순히 `is_oauth`
분기를 제거하고 일반 path 로 통합하면 안 됨 — 광고 schema 가 다른
이유가 있을 수 있음:

1. **schema 호환성**: Anthropic OAuth (Claude Code grove) 가 *알려진*
   tool name 에 대해 **자기 schema 를 강제**할 가능성. 예: 광고 schema
   와 server-side expected schema 가 다르면 4xx 또는 무시. M12 의
   `ToolSearch` 코멘트가 그 맥락을 암시함:
   > "align advertised `ToolSearch` schema with the actual dispatch
   > handler" — schema mismatch 가 실제로 문제였음.

2. **alias 매핑 의존성**: 메인 세션 system prompt / Claude-Code
   client persona 가 PascalCase 이름 (Bash, Read 등) 사용에 맞춰져
   있을 수 있음. 광고 이름 바꾸면 model 이 혼란.

3. **잘 알려진 tool 만 OAuth 가 받음** (가능성 있는 server-side
   제약). raw 이름 (`bg`, `swarm`) 보내면 4xx 거부될 위험.

→ 안전한 fix 설계: **3-Tier merge**.

### 3.2 Tier 설계 (권장 fix)

`format_tools(tools, is_oauth=true)` 새 로직:

```text
1. KNOWN_OAUTH_TOOLS (현재 10개 하드코딩) 을 base 로 둔다.
   - 이름과 schema 는 지금 그대로 유지 (안전).
   - cache_control 도 지금처럼 마지막 1개에만.

2. tools 인자 순회하며, KNOWN_OAUTH_TOOLS 에 매핑되는 이름 (alias
   매핑이 self 아닌 것) 은 skip — 이미 광고됨.

3. tools 인자 중 KNOWN 에 없는 jcode-only tool 들 추가:
   - name: tool.name (jcode 내부 이름 그대로, e.g. "bg", "swarm").
     - alias 매핑 함수가 unknown 은 그대로 통과하므로 raw 이름이
       outbound name 으로도 쓰임.
   - description: tool.description.
   - input_schema: tool.input_schema.
   - cache_control: None (마지막 1개 규칙은 KNOWN 에 이미 있음).

4. 마지막 entry 의 cache_control 을 ephemeral 로 옮긴다 (prompt
   cache 안정성 — 지금은 Write 가 마지막. fix 후에는 동적으로
   마지막이 바뀌므로 마지막 한 개에만 ephemeral 설정).
```

### 3.3 의사 코드

```rust
fn format_tools(&self, tools: &[ToolDefinition], is_oauth: bool) -> Vec<ApiTool> {
    if !is_oauth {
        // 기존 api-key path 그대로
        return tools.iter().enumerate().map(|(i, tool)| ApiTool {
            name: tool.name.clone(),
            description: tool.description.clone(),
            input_schema: tool.input_schema.clone(),
            cache_control: if i == tools.len() - 1 {
                Some(CacheControlParam::ephemeral())
            } else { None },
        }).collect();
    }

    // OAuth: 1. KNOWN_OAUTH_TOOLS hardcoded base
    let mut out: Vec<ApiTool> = vec![
        ApiTool { name: "Agent", ... },
        // ... 10개 그대로 ...
    ];
    // 마지막 entry 의 ephemeral 은 일단 비워두고 step 4 에서 설정
    for t in out.iter_mut() { t.cache_control = None; }

    // 2-3. tools 인자 중 KNOWN 에 없는 것만 추가
    let known_oauth_names: HashSet<&str> = ["Agent","Bash","Edit","Glob",
        "Grep","Read","ScheduleWakeup","Skill","ToolSearch","Write"]
        .into_iter().collect();
    for t in tools {
        let advertised = map_tool_name_for_oauth(&t.name);  // unknown → 그대로
        if known_oauth_names.contains(advertised.as_str()) {
            continue;  // 이미 광고됨
        }
        out.push(ApiTool {
            name: advertised,
            description: t.description.clone(),
            input_schema: t.input_schema.clone(),
            cache_control: None,
        });
    }

    // 4. 마지막 entry 에만 ephemeral
    if let Some(last) = out.last_mut() {
        last.cache_control = Some(CacheControlParam::ephemeral());
    }
    out
}
```

### 3.4 대안 — 더 좁은 fix (옵션 B)

만약 §4.3 검증 결과 **server-side 가 unknown tool name 거부**하면,
대안:

- **B1**: subagent 가 받는 system reminder 에 "main-session-only
  도구는 위임 금지" 명시. 메인 세션은 어차피 못 쓰는 도구 (`bg`,
  `swarm`) 가 보이지 않아 의미 없음 — **B1 단독으로는 불충분**.
- **B2**: jcode-only tool 들을 KNOWN_OAUTH_TOOLS 에 *명시적으로*
  추가하고 server-side 가 정말 거부하는지 1개 (`bg`) 부터 실험.
  거부 안 하면 §3.2 로 진행. 거부하면 server-side 가 허용하는 형태
  (예: prefix `Custom_` 같은) 찾기. 이건 §4.3 단계에 포함.

### 3.5 부수 fix — recursion guard 누수 (M43 부수 발견)

`src/tool/task.rs:430-433`:

```rust
if !allow_recursion {
    for blocked in ["subagent", "task", "todo", "todowrite", "todoread"] {
        allowed.remove(blocked);
    }
}
```

이건 `registry.tool_names()` 가 반환하는 **내부 이름** 기준. OAuth
광고 후에는 `subagent` → `Agent` 로 매핑되어 LLM 한테 `Agent` 로
보임. **현재 base path (subagent 의 base path)** 에서 `subagent` 가
allowed 에서 제거돼도, §3.2 fix 후 `tools` 인자에 `subagent` 가
이미 빠져 있으므로 자동으로 `Agent` 광고도 빠진다 — **자동 해결**.

단, **현재 (fix 전)** 는 KNOWN_OAUTH_TOOLS 가 하드코딩이라
`Agent` 가 항상 광고됨. 그래서 subagent 안에서 `Agent` 호출 시도
가능했음. §3.2 fix 후엔:

- 메인 세션: `tools` 에 `subagent` 포함 → `Agent` 광고됨 ✓
- subagent: `allowed` 에서 `subagent` 제거됨 → `tools` 에 없음 →
  `Agent` 광고 안 됨 ✓ (recursion 차단 회복)

자동으로 같이 fix 되니 별도 작업 불필요. 단 **검증 테스트는 추가**
(§5.3).

5. **prompt cache hit rate**: 광고 tool 수가 늘면 캐시 키가 바뀌니
   첫 호출은 cache miss 정상. 두 번째부터 hit 회복 확인 (provider
   metrics 또는 log).

## 5. 위험 & 완화

### 5.1 위험: server-side 가 unknown tool name 거부 (4xx)

- **확률**: 낮음 (Claude API 공식 문서상 임의 tool name 허용).
  Anthropic OAuth (Claude Code grove) 가 stricter 일 가능성은 0
  아님.
- **완화**: §4.3 실측 검증으로 선행 차단. 거부 시 옵션 B (§3.4) 로
  전환.

### 5.2 위험: prompt cache invalidation 으로 비용 증가

- **확률**: 중간. tools 가 추가되면 cache key 가 바뀜 (첫 1회만).
- **완화**: cache_control 위치 (마지막 entry) 를 일관되게 유지하면
  이후 호출은 정상 hit. `agent.rs` 의 `locked_tools` 가 세션 중간에
  tool list 가 바뀌지 않게 잠그므로 라운드 사이 캐시 hit 안정성
  유지.
- **검증**: `cargo test --lib locked_tools` 영향 없는지 확인.

### 5.3 위험: model 혼란 (이름 컨벤션 mix)

- 광고 tool 이 `PascalCase` (Bash) 와 `snake_case` (bg, swarm) 가

### 5.4 위험: subagent 의 `subagent`/`task` 차단 우회

- 현재 (fix 전): `task.rs:431` 에서 내부 이름 `subagent` 만 제거.
  KNOWN_OAUTH_TOOLS 의 `Agent` 는 무조건 광고 → **새는 중**.
- fix 후: tools 가 §3.2 로 동적이면, `tools` 에 `subagent` 없으면
  `Agent` 도 광고 안 됨 → **자동 해결**. 단 KNOWN 이 hard 광고에서
  `Agent` 를 base 로 포함하면 또 새는 거 아닌가?
  - **이 점이 §3.2 의 설계 갈림길**: KNOWN base hardcoded vs
    tools-driven 광고. 권장: **KNOWN base 도 tools 에 그 이름이
    있을 때만 포함** 으로 변경 (즉 KNOWN 은 schema 출처일 뿐
    inclusion 결정은 tools 가).
  - 의사 코드 수정:
    ```rust
    let known_schemas: HashMap<&str, (Description, Schema)> = [...];
    let mut out = vec![];
    for t in tools {
        let advertised = map_tool_name_for_oauth(&t.name);
        if let Some((desc, schema)) = known_schemas.get(advertised.as_str()) {
            out.push(ApiTool { name: advertised, description: desc.clone(),
                input_schema: schema.clone(), cache_control: None });
        } else {
            out.push(ApiTool { name: advertised, description: t.description.clone(),
                input_schema: t.input_schema.clone(), cache_control: None });
        }
    }
    if let Some(last) = out.last_mut() {
        last.cache_control = Some(CacheControlParam::ephemeral());
    }
    out
    ```
  - 이게 §3.2 보다 좀 더 정확. **fix 채택 시 이 형태 권장**.

### 5.5 위험: empty tools 인자

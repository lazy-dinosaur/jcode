## 한 줄 요약

Anthropic OAuth provider 만 hand-rolled JSON 11개로 도구를 광고함.
다른 provider (OpenAI/Gemini/Copilot/Cursor/Bedrock/Antigravity) 는
모두 `ToolDefinition.input_schema` (ToolRegistry single source of truth)
직접 직렬화. M12/M13 의 발생지가 이 hand-rolled JSON 의 stale schema
드리프트였음. 같은 문제 재발 막으려면 통일 필요.

## 핵심 파일

- **Root cause**: `src/provider/anthropic.rs` 의 `format_tools` 함수
  - `is_oauth=true` branch 의 hand-rolled JSON 11개 도구 화이트리스트
- **Single source of truth**: `src/tool/mod.rs::Registry` 의
  `ToolDefinition { input_schema: serde_json::Value }`
- **참조 패턴**: OpenAI provider 의 `format_tools` (같은 파일 또는

## 4 sub-step

### Step (a): ToolRegistry 에 OAuth 화이트리스트 정의
- `ToolDefinition` 또는 `Registry` 에 `is_oauth_safe: bool` 필드 추가
- 또는 별도 `oauth_whitelist: HashSet<String>` 를 Registry 가 보유
- 11개 도구 (현 hand-rolled JSON 의 도구 이름) 를 마킹

### Step (b): 도구 이름 매핑 자동 적용
- 현재 OAuth 는 도구 이름이 PascalCase (`Bash` 등), 다른 provider 는
  snake_case (`bash`)
- 광고 단계에서 자동 변환 함수 (`oauth_canonical_name(snake: &str)
  -> String`)
- 회귀 테스트: 모든 11개 도구 mapping 정확

### Step (c): cache_control breakpoint 로직 보존
- 현재 hand-rolled JSON 은 마지막 도구에 `cache_control` 설정
- ToolRegistry 기반 광고에도 동일 적용 (마지막 element 에 prompt
  caching breakpoint)

### Step (d): 회귀 테스트
- 광고된 schema 가 모든 `ToolDefinition.input_schema` 와 항상 일치
- snapshot test: OAuth 광고 JSON ↔ ToolRegistry 의 11개 도구 schema
- diff 0 이어야 함

## 사용자 검증 시나리오

OAuth 사용 중인 Anthropic 모델로 turn 진행 → 도구 호출 정상 동작:
- `Bash` (= bash) 도구가 OAuth 광고 schema 와 dispatch 양쪽에 일치
- 새 도구 추가 시 hand-rolled JSON 갱신 잊어버려도 자동 광고됨

## 회귀 위험

- OAuth 광고 schema 변경 → Anthropic 서버 측 캐싱 무효화 → 한 번은
  cache miss. ok, 한 번이면.
- 도구 이름 매핑 실수 → tool dispatch 실패. mitigation: snapshot test.

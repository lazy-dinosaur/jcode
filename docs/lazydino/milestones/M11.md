## 한 줄 요약

Lifecycle hook (`response.completed`, `session.stop`) 이 stdout JSON
의 `decision.action` 을 파싱하지 않음. 그래서 tool hook 의 `deny` 패턴
처럼 "AI 다음 응답에 reason 주입" 이 불가능. `tool.execute.before` 와
비대칭.

## 현재 완료된 범위 (2026-05-11)

- ✅ **Stage 1**: blocking lifecycle hook stdout decision JSON parsing.
  - 빈 stdout은 기존처럼 allow.
  - `{"action":"deny","reason":"..."}` 는 deny reason 으로 반환.
  - invalid JSON / unsupported action 은 hook failure 로 warn.
  - 여러 blocking lifecycle hook 중 deny 가 나오면 이후 hook 실행 중단.
- ✅ **Stage 2**: `response.completed` deny reason 을 다음 user turn 의 transient system reminder 로 주입.
  - `Agent::pending_lifecycle_system_reminder` runtime field 추가.
  - 기존 `system_reminder` 와 lifecycle reminder 를 병합.
  - session history 에 persist 하지 않음.
- ⚠️ `session.stop` 도 decision parsing 은 되지만, disconnect cleanup path 에서는 다음 turn 이 보장되지 않으므로 stages 3-4 에서 semantics 를 더 정리해야 함.

## 사용자 검증 시나리오

```toml
# ~/.jcode/config.toml 에 추가 (schema: [hooks] + [[hooks.commands]])
[hooks]
enabled = true

[[hooks.commands]]
event = "response.completed"
blocking = true
command = "printf '{\"action\":\"deny\",\"reason\":\"꼭 git push 하고 stop 해주세요\"}'"
timeout_ms = 3000
```

다음 turn 종료 시 hook 이 deny → 그 다음 user turn 의 system prompt 에
reason 이 transient System Reminder 로 들어가야 함.

Stage 3 loop guard 라이브 검증: 같은 deny 가 3번 연속 fire 되면 3번째에
`A lifecycle hook has denied turn completion 3 times in a row ...`
notice 가 모델에 도달하고, AI 가 정상 stop 한 뒤 streak 이 0 으로 리셋된다.

**검증 방법 주의:**
- ❌ `jcode run "..."` 를 3번 호출하면 **매 호출마다 새 Agent 프로세스** 가
  뜨므로 `lifecycle_deny_streak` 이 항상 0 에서 시작 → loop guard 트리거
  안 됨.
- ✅ **같은 Agent 인스턴스를 유지하는 path** 로 검증해야 함:
  - `jcode repl` 에 4턴을 pipe (예: `printf 'a\nb\nc\nd\n' | jcode repl`)
  - 또는 TUI 세션에서 같은 세션 안에서 짧은 메시지 4번 전송
  - 4번째 응답에서 system reminder 에 `loop guard is now suppressing` 가
    포함되어야 정상

실제 검증 결과 (2026-05-11 새벽, b78e2bf5 binary):
`jcode repl` 4턴 pipe 로 4번째 응답에서 loop guard reminder 확인됨.

## 회귀 위험

- Stage 1: 기존 lifecycle hook 사용자가 "stdout 무시되던 동작" 에
  의존 중이면 surprise. mitigation: 빈 stdout 은 여전히 allow 처리.
- Stage 3: stop hook 이 reason 으로 사용자 행동 강제 → 너무 강하면
  unstoppable AI 됨. counter + hard-stop 으로 escape 보장.

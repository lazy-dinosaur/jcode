비대칭.

## 현재 완료된 범위 (2026-05-11)

- ✅ **Stage 1**: blocking lifecycle hook stdout decision JSON parsing.
  - 빈 stdout은 기존처럼 allow.
  - `{"action":"deny","reason":"..."}` 는 deny reason 으로 반환.
  - invalid JSON / unsupported action 은 hook failure 로 warn.
  - 여러 blocking lifecycle hook 중 deny 가 나오면 이후 hook 실행 중단.
- ✅ **Stage 2**: `response.completed` deny reason 을 다음 user turn 의 transient system reminder 로 주입.
  - `Agent::pending_lifecycle_system_reminder` runtime field 추가.
  - 기존 `system_reminder` 와 lifecycle reminder 를 병합.
  - session history 에 persist 하지 않음.
- ⚠️ `session.stop` 도 decision parsing 은 되지만, disconnect cleanup path 에서는 다음 turn 이 보장되지 않으므로 stages 3-4 에서 semantics 를 더 정리해야 함.

## Stage 상태

### Stage 1: lifecycle decision parsing ✅ DONE
- 완료 commit: `e345d632` / source `71101b8e`
- `run_blocking_lifecycle_hook` 이 stdout JSON 을 파싱하고 deny reason 을 반환
- 회귀 테스트 추가: invalid JSON, unsupported action, missing reason fallback, deny 후 후속 hook 중단

### Stage 2: reason → next-turn system reminder inject ✅ DONE
- 완료 commit: `e345d632` / source `71101b8e`
- `response.completed` deny reason 을 다음 user turn 의 transient system reminder 로 주입
- 기존 current-turn system reminder 와 병합, session history 에 persist 하지 않음

### Stage 3: `stop_hook_active` 무한루프 방지
- N회 연속 deny 시 hard-stop
- 카운터를 session state 에 저장 (예: `stop_hook_consecutive_deny: u8`)
- threshold (예: 3) 넘으면 reason 무시하고 진짜 stop
- 회귀 테스트: stop hook 이 3번 연속 deny → 4번째에 강제 stop

### Stage 4: `session.stop` vs `client.disconnect` 의미 분리 (M12 후속)
- 사용자 명시 stop 과 client 비정상 disconnect 를 분리
- hook 이 둘 다에 같은 동작하지 않게

### Stage 5: payload context 보강 ★ 핵심
- 현재 hook script 는 빈 stdin 으로 호출 → 검증 불가능
- 추가 필드: `last_user_message`, `recent_tool_calls`,
  `turn_count`, `session_age_seconds`
- hook script 가 "마지막 메시지가 X 면 deny" 같은 정책 작성 가능
- JSON payload schema 정의 + 회귀 테스트

```bash
'''

# 다음 turn 종료 시 hook 이 deny → 그 다음 user turn 의 system prompt 에

```

`session.stop` hook 도 decision JSON parsing 은 되지만, disconnect cleanup 은
다음 turn 이 보장되지 않으므로 stage 4 에서 semantics 를 더 정리한다.

## 회귀 위험

- Stage 1: 기존 lifecycle hook 사용자가 "stdout 무시되던 동작" 에
  의존 중이면 surprise. mitigation: 빈 stdout 은 여전히 allow 처리.
- Stage 3: stop hook 이 reason 으로 사용자 행동 강제 → 너무 강하면
  unstoppable AI 됨. counter + hard-stop 으로 escape 보장.

## TL;DR (post-fix)

**Not actually a hang.** The `thinking… 108.3s · checking websocket ·
existing websocket · +1 queued` display surfaced a real long-running
`bg` tool (91.68s observed in logs). The transport was healthy the
whole time. The misleading label was a stale `StatusDetail` payload
that nothing cleared after the websocket healthcheck succeeded.

Three small changes give `StatusDetail` an explicit `""` = clear
contract, mirrored on the provider, agent, and TUI sides.

## Root cause (확정)

`StatusDetail { detail: String }` 에는 clear semantics 가 없었음:

- emit: `emit_status_detail(tx, "checking websocket")` → `Some(detail)` set
- healthcheck 성공해도 overwrite/clear 안 함
- Thinking 으로 넘어가도 `append_transport_context()` 가 마지막 detail
  계속 렌더 (`src/tui/ui_input.rs:523-525, 338-346`)

라이브 로그가 hang 가설을 배제:
```
00:00:39.091 Persistent WS idle for 28624ms; sending healthcheck ping before reuse
00:00:39.242 Persistent WS healthcheck pong after 151ms
00:00:39.243 Persistent WS continuation request sent
00:00:39.829 Persistent WS first server event after 579ms
00:00:47.720 donkey API call complete
00:00:56.481 donkey Tool starting: bg
00:02:28.158 donkey Tool finished: bg in 91.68s
```
즉 hang 이 아니라 `bg` tool wait + stale UI detail (시간 규모 일치).

진단 보고서는 라이브 세션의 searcher subagent (`397216nb24`) 결과.

## Fix summary

| 파일 | 변경 |
|---|---|
| `src/provider/openai_stream_runtime.rs` | healthcheck 성공 직후 `emit_status_detail(tx, "")` 호출 (clear) |
| `src/tui/app/remote/server_events.rs` | 빈 detail 받으면 `app.status_detail = None` |
| `src/agent/turn_streaming_mpsc.rs` | agent-side `last_status_detail` 도 동일 contract (resume/recovery 보호) |
| `src/tui/app/tests/remote_startup_input_04.rs` | 회귀 테스트 `test_handle_server_event_empty_status_detail_clears_label` 추가 |

기존 `"reusing websocket"` / `"opening websocket"` overwrite path 는
non-empty payload 라 그대로 동작.

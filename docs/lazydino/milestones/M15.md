## 핵심 파일 + Drop Candidate 3개

### Candidate A: `HistoryMessage` 에 image 필드 추가
- 위치: `crates/jcode-protocol/src/lib.rs:47-56`
- 현재 `HistoryMessage.content` 는 text-only
- image bytes 가 sibling `ServerEvent::History.images: Vec<RenderedImage>`
  로만 흐름
- 영향 범위: protocol 변경 → 모든 client 호환성 영향

### Candidate B: `Client::get_history()` 가 images 필드 직접 포함 ★ 최소 fix
- 위치: `src/server/client_api.rs:154-159`
- 현재 `Client::get_history()` 가 `images` 필드 명시적으로 버림
- 이걸 보존하는 게 가장 작은 fix
- M18 의 `get_history_with_images()` 와 비슷한 패턴 적용 가능
- **추천**: 여기부터 시작

### Candidate C: live attached client 에 fanout
- 위치: `src/server/client_lifecycle.rs:2658-2669`
- 라이브 attached 형제 client 에 user-message-with-images 이벤트
  fanout 안 함
- B 가 끝나야 의미 있음

## 회귀 위험

- protocol 변경 없으니 client 호환성 OK (Candidate B 만 진행 시)
- image bytes 크기 → network 부하. 큰 이미지 한 번에 보내면 메모리
  spike. mitigation: chunked transfer 는 별 milestone 으로.

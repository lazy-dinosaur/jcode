# M48 - Opencode-style durable compaction pipeline

**Priority**: High (long-context reliability, overflow recovery, session replay quality)
**Status**: In Progress (C-0, C-1, C-2, C-3, C-4a, C-5a, C-6a, C-7a done 2026-05-17)
**Discovered**: 2026-05-17 during opencode source comparison after a large Jcode session hit emergency compaction.
**Custom patch type**: Lazydino custom patch stack. Upstream PR only after the storage and provider abstractions settle.

## One-line summary

Move Jcode compaction from an in-memory manager plus stored summary state toward an opencode-style durable compaction turn: a first-class marker, an assistant summary artifact, token-budgeted recent-tail preservation, tool-output pruning, and controlled post-compaction continuation.

## Source comparison

### Opencode reference points

Files inspected under `/tmp/opencode-src/packages/opencode/src`:

- `session/overflow.ts`
  - `usable()` computes effective input budget as context minus reserved output budget.
  - `isOverflow()` only triggers when compaction auto mode is enabled and tokens reach usable budget.
- `session/compaction.ts`
  - Constants: `PRUNE_MINIMUM = 20000`, `PRUNE_PROTECT = 40000`, `TOOL_OUTPUT_MAX_CHARS = 2000`, `DEFAULT_TAIL_TURNS = 2`, recent token budget clamp `2000..8000`.
  - `create()` inserts a user message with a `compaction` part.
  - `process()` writes an assistant message with `mode = "compaction"`, `agent = "compaction"`, `summary = true`, no tools, and stripped media.
  - `completedCompactions()` links user compaction marker to assistant summary by parent id and hides old compaction pairs from later summaries.
  - `select()` keeps a token-budgeted tail by recent turns and can split within a turn if the whole turn is too large.
  - `prune()` marks older completed tool outputs compacted after protecting recent turns and `PRUNE_PROTECT` tokens.
  - Overflow path can replay the previous real user message with media replaced by text labels, or synthesize a continuation prompt.
- `session/summary.ts`
  - Session diff summary is separate from compaction but uses the same message/part durability model.

### Current Jcode reference points

Files inspected in this repository:

- `src/compaction.rs`
  - `CompactionManager` tracks `compacted_count`, `active_summary`, background task handle, token estimates, proactive/semantic modes, and failure streak gates.
  - Thresholds come from `jcode-compaction-core`: default 200k budget, 80 percent background trigger, 95 percent emergency hard compact, keep 10 recent turns, minimum keep 2.
  - OpenAI native compaction is represented by `ContentBlock::OpenAICompaction` and persisted in session compaction state.
- `src/agent.rs::messages_for_provider`
  - Takes the session provider messages, runs `ensure_context_fits()` only when `provider.uses_jcode_compaction()`, then asks the manager for API messages.
  - Applies compaction events by resetting provider session/cache state and persisting `Session.compaction`.
- `src/agent/compaction.rs`
  - Manual compaction and context-limit auto-recovery are manager calls, not durable transcript turns.
  - Context-limit recovery hard-compacts and retries when provider error text looks like overflow.
- `src/session.rs`
  - `messages_for_provider()` is a cached transform from stored messages. Compaction is separate metadata, not a message pair.
- `crates/jcode-message-types/src/lib.rs`
  - Provider-visible content blocks have `Text`, `Reasoning`, `ToolUse`, `ToolResult`, `Image`, and `OpenAICompaction`.

## Key gaps

1. Compaction is not a durable conversation turn.
   - Jcode stores a compacted state and injects a synthetic summary block into provider payloads.
   - Opencode stores both the compaction marker and assistant summary as normal session artifacts.

2. Recent-tail preservation is count-based, not token-budgeted.
   - Jcode keeps 10 recent turns by default.
   - Opencode keeps a small recent tail by token budget, defaulting to a 2-turn goal and splitting large turns.

3. Tool-output pruning is not a first-class safe pass.
   - Jcode has emergency truncation helpers.
   - Opencode has a normal prune pass that protects recent context and marks older tool outputs compacted.

4. Overflow recovery loses intent in some cases.
   - Jcode hard-compacts and retries the same turn after context-limit errors.
   - Opencode can replay the previous real user message or synthesize a continuation prompt after compaction.

5. OpenAI native compaction must stay compatible.
   - Jcode already has a valuable encrypted-content path. M48 must preserve it and define how native artifacts coexist with durable compaction turns.

## Goals

1. Represent each compaction as durable session data that can be searched, rendered, exported, and replayed.
2. Preserve recent user intent with token-budgeted tail selection, not only fixed turn count.
3. Add safe tool-output pruning before full summary compaction when it is enough.
4. Make overflow recovery explicit: marker, summary, replay or synthetic continue, retry telemetry.
5. Keep existing `CompactionManager` behavior available behind a compatibility path during migration.
6. Preserve OpenAI native encrypted compaction and text fallback semantics.
7. Add deterministic fixtures so compaction decisions can be unit tested without live providers.

## Non-goals

- Removing proactive or semantic compaction modes in this milestone.
- Changing memory extraction semantics except to ignore durable compaction internals correctly.
- Replacing provider-native OpenAI compaction.
- Reworking interrupt/cancel lifecycle. That is M49.

## Proposed data model

Prefer a session-local durable model that does not send marker internals to providers by default.

Option A, recommended:

- Add `StoredCompactionTurn` metadata to `StoredMessage` or a sidecar map keyed by message id:
  - `kind = "marker" | "summary"`
  - `auto: bool`
  - `overflow: bool`
  - `tail_start_id: Option<String>`
  - `previous_summary_id: Option<String>`
  - `summary_of_message_ids: Vec<String>` or a compact range descriptor
- Store the marker as a user-role stored message with no provider-visible content.
- Store the summary as an assistant-role stored message with provider-visible text only when building the compacted provider payload.

Option B:

- Add a new hidden `ContentBlock::CompactionMarker` variant.
- Every provider adapter must ignore it. This is more invasive because `ContentBlock` is shared with provider payload conversion.

M48 should start with Option A unless implementation proves sidecar metadata is harder than hidden content blocks.

## Provider payload contract

After successful durable compaction, `messages_for_provider()` should build:

1. A synthetic user or system-reminder style text block containing the latest durable summary.
2. Optional `ContentBlock::OpenAICompaction` when the active provider is OpenAI and the encrypted content is sendable.
3. A recent tail beginning at `tail_start_id`.
4. No compaction marker internals.
5. No old compaction assistant summaries except the chosen anchor summary.

The provider payload must be stable across repeated calls when no new session messages are added, to protect prompt cache behavior.

### C-2. Token-budgeted selection algorithm

Branch: `patch/m48-c2-token-tail-selection` — **DONE 2026-05-17** (merged into `deploy/m9-m27-catchup` as `e927e619`).

Implemented in `jcode-compaction-core::m48_select`:

- `usable_budget(ctx, reserved)` with saturating subtraction; default reserved = `COMPACTION_BUFFER = 20_000`.
- `preserve_recent_budget(usable, override)` — override wins, otherwise `clamp(floor(usable / 4), 2_000, 8_000)`.
- `turns(messages)` — walks user-led turn boundaries, skips compaction markers.
- `select_tail(messages, budget, tail_turns_limit)` — preserves the last `tail_turns_limit` turns (default 2) under the budget; calls `split_turn` mid-turn when the next whole turn would overflow.
- `split_turn(messages, turn, budget)` — scans forward for the first message index whose `[i..end)` suffix fits.

`CompactionConfig` now carries `auto`, `prune`, `tail_turns`, `preserve_recent_tokens`, `reserved_tokens` (all `#[serde(default, skip_serializing_if=...)]`).

Validation: 11 new unit tests in `m48_select::select_tests` plus a property-style fallback test (oversized single turn → "summarize everything"). 30 / 30 `jcode-compaction-core` lib tests pass. No runtime caller wired yet; C-3 will attach `select_tail` to the prune path.

### C-3. Tool-output pruning pass

Branch: `patch/m48-c3-tool-output-prune` — **DONE 2026-05-17** (merged into `deploy/m9-m27-catchup` as `0cdffd3d`).

Implemented in `jcode-compaction-core::m48_prune`:

- `prune(messages, protected_tools, protect_recent_turns, prune_protect_tokens, prune_minimum_tokens)` walks backwards, protects the last `protect_recent_turns` user-led turns (default 2), accumulates `ToolResult.content.len()`, and once the rolling total passes `PRUNE_PROTECT` (40k) marks every later `ToolResult` for replacement with `PRUNED_PLACEHOLDER`. Only commits when `bytes_recovered > PRUNE_MINIMUM` (20k).
- Tool-result-only user messages do NOT count as new turns, so the multi-message-per-turn shape `user text + assistant tool_use + user tool_result` collapses to a single turn boundary.
- Tools in `DEFAULT_PROTECTED_TOOLS = ["skill"]` never accumulate into the rolling budget and are never pruned.
- Idempotent: `PRUNED_PLACEHOLDER` payloads are skipped on subsequent passes.
- Pure-functional API returning `(Vec<Message>, PruneReport)`. Persistence wiring is M48-C4's responsibility.

Validation: 6 new unit tests in `m48_prune::prune_tests` cover empty input, large output pruning + protect window, protected-tools-never-pruned, idempotence, protect_recent_turns hard skip, rolling-budget tail protection. 36 / 36 `jcode-compaction-core` lib tests pass.

#### C-6a. Native vs text precedence helpers — **DONE 2026-05-17**

Branch: `patch/m48-c6-native-coexistence` (merged into `deploy/m9-m27-catchup` as `ccac7ee5`).

Implemented in `jcode-compaction-core::m48_native`:

- `enum ProviderKind { OpenAIResponses, Anthropic, Gemini, OpenRouter, Other }` with `supports_native_encrypted_content()` and `classify_provider_id(id)` lookup.
- `enum SummaryRepresentation { Native { len }, Text { dropped_native_len: Option<usize> }, None }` for telemetry-safe callers.
- `decide_summary_representation(provider, encrypted_content, text_summary, safe_max_chars)` implements the precedence:
  1. Non-OpenAI providers → text (or None).
  2. OpenAI + blob fits → Native(len). Text suppressed in payload.
  3. OpenAI + oversized blob + text → Text { dropped_native_len: Some(len) } so the call-site can log it.
  4. OpenAI + oversized blob + no text → None (caller must resend head or recompact).
  5. OpenAI + no blob + text → Text { dropped_native_len: None }.
  6. Whitespace-only text summaries are ignored.
- `provider_can_consume_blob(provider)` for the cross-provider failover path.

Validation: 10 new unit tests; 67 / 67 `jcode-compaction-core` lib tests pass; `cargo check -p jcode` clean. `safe_max_chars` is injected by callers (today `OPENAI_ENCRYPTED_CONTENT_SAFE_MAX_CHARS` from `jcode-provider-openai::request`) so `jcode-compaction-core` stays provider-agnostic.

## Risk register

- Provider cache regression: durable summary injection can change prefix shape. Mitigation: cache-stability tests and exact provider payload snapshots.
- Session schema complexity: hidden marker content may leak to providers. Mitigation: prefer sidecar metadata and explicit provider conversion tests.
- Native OpenAI encrypted content size: encrypted blobs can exceed request limits. Mitigation: keep existing sendability gate and fallback text summary.
- Infinite compaction loops: preserve M14 failure streak gate and add overflow retry counters.
- Search/export noise: marker messages can pollute user-visible history. Mitigation: render/export/search filters.

## Done criteria

1. Compaction creates durable marker and summary artifacts in session storage.
2. Provider payload includes exactly one active summary plus a token-budgeted recent tail.
3. Tool-output prune pass reduces provider payload while preserving transcript structure.
4. Overflow recovery can compact and continue without repeating runaway hard compaction.
5. OpenAI native encrypted compaction remains supported with text fallback.
6. Targeted tests cover all stage fixtures and old session compatibility.
7. Release build succeeds.

# M46 — selfdev reload activation/idempotency 안정화

**Priority**: High (self-dev 배포 신뢰성 / live debugging UX)
**Status**: Planned
**Discovered**: 2026-05-13 live M45 rollout 중 user report: “리로드 시도할때 여러번 리로드되거든??”

## 한 줄 요약

`selfdev reload`가 한 번의 배포 의도에서 여러 번 interruption/reconnect를 만들고,
빌드 산출물은 만들어졌는데 `current` / `shared-server` channel activation 이 기대대로
바뀌지 않는 듯 보이는 문제를 분리해서 안정화한다.

## 관측된 증상

- `selfdev build`는 `~/.jcode/builds/versions/<hash>/jcode` 산출물을 정상 생성.
- `selfdev reload` tool call은 intentional interruption 으로 output missing 이 발생.
- 이후 `selfdev status`에서:
  - recent builds에는 새 hash가 보임.
  - 하지만 `Current` / `Shared server`가 이전 hash로 계속 보일 수 있음.
  - `Pending activation` / `SocketReady` 상태가 남거나, 사용자가 여러 번 reload 된 것처럼 보임.
- debug socket session registration 이 reload 직후 순간적으로 현재 세션을 못 찾는 경우가 있음.

## 의심 코드 경로

- `src/tool/selfdev/reload.rs`
  - `publish_local_current_build_for_source`
  - `update_shared_server_symlink`
  - `manifest.set_pending_activation`
  - AgentTurn mode 의 intentional infinite wait/interruption path
- `src/tool/selfdev/status.rs`
  - `env!("JCODE_VERSION")` 기반 Running 표시와 build channel 표시가 혼재.
- `src/server/reload.rs`, `src/server/reload_state.rs`
  - reload signal ack, socket ready handoff, recovery marker lifecycle.

## 성공 기준

1. 한 번의 `selfdev reload` 요청은 한 번만 server reload/handoff를 유발한다.
2. reload 이후 `current`와 `shared-server` channel 이 target hash를 정확히 가리킨다.
3. `selfdev status`가 다음을 구분해서 표시한다:
   - running binary hash
   - current channel hash
   - shared-server channel hash
   - pending activation target / owner
4. AgentTurn reload interruption 이 사용자 화면에 중복 완료/중복 reload처럼 보이지 않는다.
5. regression test:
   - build artifact exists but channel not advanced 상태 재현
   - pending activation completion/rollback idempotency
   - repeated reload call does not spawn duplicate activation for same request

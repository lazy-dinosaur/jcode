# M45 — Private `.jcode/` instruction discovery 강화

**Priority**: High (private instruction-following / monorepo UX)
**Status**: 🟡 IN PROGRESS — Stage 1/2 implemented and validated 2026-05-13; Stage 3 remains
**Discovered**: 2026-05-13 live user report: “jcode 는 agents.md 를 크게 읽어서 따라가지 않는것같은데?? 구조를 opencode 같은거랑 비교해볼까??”
**Correction**: 2026-05-13 user clarified: “jcode 를 프라이빗 하게 사용하고싶어서 호환성은 중요하지 않아”

## 한 줄 요약

jcode 를 개인용 harness 로 쓸 때, 공개 repo 의 team `AGENTS.md` 보다
private `.jcode/` 아래의 상세 지침/모듈/monorepo package별 규칙을 더 강하게,
더 명확하게 읽고 따르게 만들 필요가 있다. opencode 비교는 참고 자료일 뿐이고,
목표는 호환성이 아니라 **gitignored private instruction stack** 의 품질이다.

## 현재 jcode 코드 근거

- `src/prompt.rs::build_system_prompt_{full,split}`
  - `load_agents_md_files_from_dir(working_dir)` 결과를 system prompt 에 추가.
- `src/prompt.rs::load_agents_md_files_from_dir_with_config`
  - project: `<project>/AGENTS.md` 만 읽음.
  - global: `~/.AGENTS.md` 만 읽음.
  - private: `<project>/.jcode/AGENTS.md` 만 읽음.
  - project dir 탐색 기준: ancestor 중 `.jcode` 또는 `AGENTS.md` 가 있는 첫 위치.
- `src/prompt.rs::load_prompt_overlay_files_from_dir_with_config`
  - `~/.jcode/prompt-overlay.md`, `<project>/.jcode/harness/*.md`,
    `<project>/.jcode/prompt-overlay.md` 는 지원.
- 현재 부족한 것:
  - `.jcode/AGENTS.md` 외 private instruction 파일을 명시적으로 여러 개 로딩하는 config.
  - `.jcode/harness/*.md` 는 있지만 “항상 읽어야 하는 private rules” 와
    “필요할 때 읽는 상세 docs” 의 구분/가시성 부족.
  - monorepo package/subdir 별 private rules (`packages/foo/.jcode/AGENTS.md` 등)
    탐색/주입 부재.
  - `/info` 에 loaded instruction **path/title/order** 가 충분히 드러나지 않아
    실제로 무엇을 읽었는지 확인하기 어려움.
  - 파일 read 시 해당 파일 주변의 private nested instruction 을 turn 에 추가 주입하는 기능 부재.

참고: jcode 의 **agent profile markdown** 은 별개로
`.jcode/agents`, `.claude/agents`, `.agents/agents`, `.opencode/agents` 를 읽는다
(`src/agent_profiles_md.rs`). 이것은 subagent/profile 정의이지, general project
instructions (`AGENTS.md`) 로딩과는 다른 경로다.

## opencode 비교에서 참고할 점 (호환 목표 아님)

opencode `packages/opencode/src/session/instruction.ts` 기준으로 참고할 만한 구조:

- `systemPaths()`:
  - config `instructions`: local glob, absolute path, `~/...`, URL 지원.
- `system()`:
  - 모든 instruction file/URL content 를 `Instructions from: <path>` 로 system context 에 합침.
- `resolve(messages, filepath, messageID)`:
  - read tool 이 파일을 읽을 때, 그 파일의 directory 에서 project root 까지 올라가며
    nearby instruction file 을 찾아 **한 assistant message 당 한 번** attach.
  - 이미 system instruction 이거나 이미 loaded 된 file 은 dedup.

여기서 가져올 핵심 아이디어는 “호환 파일명” 이 아니라:
1. 여러 instruction source 를 path label 과 함께 합친다.
2. custom instruction globs 를 지원한다.
3. file-local nested instruction 을 필요할 때 주입하고 dedup 한다.

## 사용자 영향

- private `.jcode/AGENTS.md` 를 길게 써도 실제 prompt 에 어떻게 들어갔는지 확인하기 어려움.
- private 상세 지침을 여러 파일로 쪼개고 싶어도 `.jcode/harness/*.md` 외에는
  명시적인 include/glob mechanism 이 부족함.
- monorepo package-specific private rule 을 두고 싶어도 root 기준 한 벌만 주입되기 쉬움.
- “내 개인 jcode harness 는 이렇게 동작해라”가 team repo 파일과 섞여 우선순위가 흐려질 수 있음.

## 권장 scope

### Stage 1 — private instruction visibility + priority 정리

1. `/info` prompt accounting 에 loaded instruction path/name/order 표시.
2. `.jcode/AGENTS.md` 와 `.jcode/harness/*.md` 를 “private highest-priority” 로 명확히 label.
3. `ignore_project_agents = true` 사용 시 team `<project>/AGENTS.md` 를 확실히 빼고
   private `.jcode/*` 만 남는지 regression test 강화.
4. 현재 prompt order 문서화:
   - base prompt
   - team project AGENTS (optional)
   - global AGENTS (optional)
   - private `.jcode/AGENTS.md`
   - private `.jcode/harness/*.md` / overlay

**2026-05-13 implementation note**:

- Added ordered `PromptInstructionSource` metadata to `ContextInfo`.
- `/info` and `/context` now display loaded/skipped instruction sources with
  status, private/shared visibility, char counts, skip reason, and path.
- `ignore_project_agents=true` now reports team `AGENTS.md` as skipped while
  preserving private `.jcode/AGENTS.md` visibility.
- Private `.jcode/harness/*.md` modules report deterministic loaded order.
- Validation: `cargo +nightly test --lib prompt_tests -- --nocapture`,
  `cargo +nightly check --lib`, `git diff --check`, and self-dev build passed.

### Stage 2 — private custom instruction paths

1. `[prompt] private_instructions = [...]` 또는 `[prompt] instructions = [...]` config 추가.
   - relative glob, absolute path, `~/...` 지원.
   - 기본 권장 root 는 `<project>/.jcode/`.
   - URL 은 private usage 목표와 맞지 않으므로 초기 non-goal.
2. loaded content 를 `Instructions from: <path>` 형태로 label.
3. tests: glob ordering, missing file tolerant, duplicate path dedup.

**2026-05-13 implementation note**:

- Added `[prompt] private_instructions = [...]`.
- Relative paths resolve under `<project>/.jcode/`; `.jcode/...`, absolute paths,
  and `~/...` are also accepted.
- Supports files and local glob patterns, deterministic sorted load order, missing
  tolerant expansion, and dedup against `.jcode/harness/*.md` / overlays.
- Sources are labeled as `Private Jcode Instruction (<path>)` and appear in
  `/info` / `/context` through Stage 1 source reporting.
- Validation: prompt/config focused tests, `cargo +nightly check --lib`,
  `git diff --check`, and self-dev build/reload passed.

### Stage 3 — nested private instruction injection

1. Read/grep/edit 등 file-local tool 이 특정 파일을 load 할 때,
   해당 path 주변의 nearest private instruction file 을 context 에 주입하는 mechanism 설계.
   후보 파일명:
   - `.jcode/AGENTS.md`
   - `.jcode/instructions.md`
   - `.jcode/rules/*.md`
2. message/turn scoped dedup 필요.
3. 위험: 매 tool read 마다 system context 증가 → token budget 관리 필요.
   - first implementation 은 “system reminder once per turn” or “tool result metadata + next turn inject” 중 선택.

## Verification checklist

- Unit tests:
  - private `.jcode/AGENTS.md` 가 team `AGENTS.md` 뒤에 로드되어 우선순위를 가짐.
  - `ignore_project_agents=true` 일 때 private `.jcode/*` 는 유지되고 team AGENTS 만 빠짐.
  - private instruction glob load + deterministic order + dedup.
  - nested private `.jcode/AGENTS.md` find-up / nearest-first 정책.
- Integration/manual:
  - root-only AGENTS 기존 동작 회귀 없음.
  - private `.jcode` only repo 에서 `/info` 에 loaded path/order 표시.
  - monorepo package-local private rule 을 read 후 모델이 해당 규칙을 인지하는지 확인.

## Non-goals / caution

- opencode/Claude compatibility 를 목표로 삼지 말 것.
- remote URL instruction 은 private usage 목표와 맞지 않으므로 기본 non-goal.
- nested instruction auto-inject 는 token 폭증 위험이 있어 Stage 1/2 이후 별도 검증.

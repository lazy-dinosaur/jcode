## 구현 후보 3가지

### (a) `jcode build-cleanup --keep N` 보조 CLI ★ 추천
- jcode binary 자체에 cleanup 서브커맨드 추가
- `~/.jcode/builds/versions/` 스캔
- mtime 기준 최근 N 개 + stable/current 가 가리키는 것만 보존
- 나머지 삭제
- 작업량: ~80줄

### (b) Atomic deploy 스크립트 안에서 자동 cleanup
- 빌드 + 배포 시점에 자동으로 옛 versions 정리
- 위험: 사용자가 옛 binary 로 rollback 하려는 의도 무시
- 비추천

### (c) Systemd timer 로 주기적 cleanup
- ~/.config/systemd/user/jcode-cleanup.timer
- daily 실행
- 작업량: ~30줄 (shell script + timer unit)

**추천: (a)**. 명시 실행이 안전하고 다른 머신/사용자도 활용 가능.

## 작업 분해 (옵션 a)

1. CLI subcommand 등록 (`src/cli/`): ~20줄
2. versions/ 스캔 + 보존 정책: ~30줄
3. 안전장치 (current/stable 가리키는 거 절대 안 삭제): ~20줄
4. 회귀 테스트 (tmp dir 에 fake versions 만들고 cleanup): ~30줄

총 ~100줄, ~1-2h.

## 사용자 검증 시나리오

```bash
# 현재 상태
ls ~/.jcode/builds/versions/ | wc -l  # 예: 17

# cleanup
jcode build-cleanup --keep 3

# 결과: 3개만 남아야 (current + stable + 최근 1개 = ~3개)
ls ~/.jcode/builds/versions/  # 예: 3
```

## 회귀 위험

- 옛 binary 로 rollback 하고 싶은 경우 cleanup 너무 공격적이면
  곤란. mitigation: `--dry-run` 옵션, `--keep` default 5+
- 떠 있는 jcode process 가 옛 binary 의 path 를 inode 로 들고 있을
  수 있음. fs 삭제는 ok 지만 디스크 회수는 process 종료 후. OK.

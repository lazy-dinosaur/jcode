# M49 - Opencode-style cooperative interrupt and turn lifecycle

**Priority**: High (safe stop behavior, provider/tool cleanup, transcript integrity)
**Status**: In Progress (C-0/C-1/C-2/C-3/C-4 done 2026-05-18)
**Discovered**: 2026-05-17 during opencode interrupt and abort lifecycle comparison.
**Custom patch type**: Lazydino custom patch stack. Candidate upstream PR after cooperative cancellation proves stable.

## One-line summary

Replace Jcode's hard task-abort interrupt path with an opencode-style cooperative turn control plane: typed abort reasons, provider/tool cancellation propagation, partial transcript finalization, idempotent UI/server abort, and clean separation from reload handoff.

## Source comparison

### Opencode reference points

Files inspected under `/tmp/opencode-src/packages/opencode/src`:

- `cli/cmd/tui/component/prompt/index.tsx`
  - `session.interrupt` is a two-step command. First press arms the interrupt, second press calls `sdk.client.session.abort({ sessionID })`.
- `server/routes/instance/httpapi/groups/session.ts`
  - Exposes `session.abort` as a server control endpoint.
- `cli/cmd/run/runtime.ts`
  - Tracks an `aborting` boolean during model/session switching so repeated aborts are idempotent.
- `cli/cmd/run/runtime.queue.ts`
  - Each queued run owns an `AbortController` and aborts active work when the queue closes or switches.
- `cli/cmd/run/stream.transport.ts`
  - Races turn completion against abort signals.
  - Uses stream `takeUntil` behavior for footer close, turn abort, or external abort.
  - Flushes `turn.abort` or `turn.cancel` so in-flight parts are finalized as interrupted.
- `cli/cmd/run/session-data.ts`
  - `failTool()` emits interrupted final entries for in-flight tool parts when a turn is aborted.
- `background/job.ts`
  - Uses Effect fiber interruption and cleanup to stop background work.

### Current Jcode reference points

Files inspected in this repository:

- `src/server/client_lifecycle.rs::cancel_processing_message`
  - Removes the processing task handle, calls `session_control.request_cancel()`, waits 500 ms, then calls `JoinHandle::abort()` and waits up to 2 s.
  - Sends `ServerEvent::Interrupted` and `ServerEvent::Done` after cancellation.
- `src/server/state.rs::SessionControlHandle`
  - Holds `soft_interrupt_queue`, `background_tool_signal`, and `stop_current_turn_signal`.
  - `stop_current_turn_signal` is wired from `Agent::graceful_shutdown_signal()` at session setup.
- `crates/jcode-agent-runtime/src/lib.rs::InterruptSignal`
  - Atomic flag plus `tokio::Notify`; good primitive, but currently reasonless.
- `src/agent/turn_streaming_mpsc.rs`
  - Single-tool path races tool completion against `background_tool_signal` and `graceful_shutdown`.
  - Reload can abort or wait briefly for bash handoff.
  - User urgent soft interrupts skip remaining tools before execution.
- `src/agent/turn_execution.rs::dispatch_tools_parallel`
  - Parallel tool dispatch has a `CancellationToken`, but it is only cancelled after urgent soft interrupts are detected while collecting results.
- `src/agent/interrupts.rs`
  - Soft interrupts are queued out of band and injected at safe points B/D.
  - Background tool signal and graceful shutdown signal are exposed separately from the agent lock.

## Key gaps

1. User cancel and reload shutdown share a signal path.
   - `stop_current_turn_signal` is currently built from `Agent::graceful_shutdown_signal()`.
   - Tool code that sees the signal cannot reliably know whether this is user interrupt, client disconnect, or server reload.

2. Provider calls are not cooperatively cancelled.
   - The server aborts the Tokio processing task after 500 ms.
   - Provider HTTP/WebSocket streams do not receive a typed cancellation token through the provider trait.

3. Tool cancellation is uneven.
   - Single-tool mpsc path supports Alt+B background handoff.
   - Parallel dispatch has a cancellation token, but it is not connected to the server cancel signal.
   - Bash/process cleanup depends on tool-specific graceful shutdown handling.

4. Transcript finalization is incomplete under abort.
   - Hard abort can drop partial assistant text or in-flight tool state before a consistent message/result is persisted.
   - Opencode explicitly flushes interrupted parts.

5. Abort UX is not fully idempotent.
   - Server cancellation checks `state.task`, but there is no typed per-turn lifecycle state exposed to the UI/debug socket.
   - Repeated interrupts can race with task completion and reset signals.

6. Follow-up captured during M48-C4b: screenshot/image rendering bug.
   - User reported an attached screenshot/image bug on 2026-05-18 while C4b was in progress.
   - Before starting M49 implementation, inspect the session transcript attachment and decide whether this belongs in M49 diagnostics/interrupt UI, M48 media compaction follow-up, or a separate visual rendering milestone.

## Goals

1. Introduce a typed turn cancellation model separate from graceful reload shutdown.
2. Propagate cancellation to provider streams, tool execution, bash child processes, and background adoption logic.
3. Persist partial assistant and tool results as interrupted artifacts instead of losing them.
4. Make server/TUI interrupt idempotent and observable.
5. Preserve soft interrupts and Alt+B background behavior while making both reason-aware.
6. Keep reload handoff semantics for self-dev and long bash tools.

## Non-goals

- Replacing soft interrupts. They remain the safe-point mechanism for adding user/system messages during a live turn.
- Rewriting every provider adapter in one patch. The provider cancellation surface can land first with no-op defaults.
- Changing M48 compaction behavior.

## Proposed core design

Add a dedicated turn control object with typed reasons.

```rust
pub enum TurnStopReason {
    UserInterrupt,
    ClientDisconnect,
    ServerReload,
    BackgroundCurrentTool,
    Superseded,
}

pub struct TurnControl {
    pub stop: InterruptSignal,
    pub reason: Arc<RwLock<Option<TurnStopReason>>>,
    pub tool_cancel: CancellationToken,
}
```

Guidelines:

- `UserInterrupt` must not be treated as `ServerReload`.
- `ServerReload` may preserve the existing bash/selfdev handoff behavior.
- `BackgroundCurrentTool` should only detach the selected tool, not cancel the whole provider turn unless required.
- `ClientDisconnect` should be configurable: continue unattended or stop, depending on session mode.
- Signals should be one-shot per turn and replaced at every new turn start.

### C-5. Server lifecycle idempotency

Branch: `patch/m49-c5-server-cancel-idempotent`

- Replace `cancel_processing_message()` hard-coded timings with a cooperative sequence:
  1. Mark turn state as `Cancelling(reason)`.
  2. Fire turn control.
  3. Wait cooperative grace period from config, default 1500 ms.
  4. Abort task only as fallback.
  5. Finalize transcript and emit one `Interrupted` plus one `Done`.
- Ignore repeated cancel requests for the same turn after state enters cancelling, but ack them to the client.
- Reset turn control only after the processing task has fully observed completion/cancellation.

Validation:

- Double interrupt produces one transcript finalization and one Done.
- Cancel racing with normal completion does not emit duplicate Done.
- Background swarm status moves to stopped exactly once.

## Interactions with M48

- M48 may add durable compaction turns. M49 must not treat compaction marker/summary generation as a normal user-visible turn when aborting.
- If a compaction summary generation is cancelled, it should leave either no durable summary or a summary message marked interrupted. It must not update the active compaction pointer.
- M49 transcript finalization helpers should be reusable by M48 overflow replay if an overflow retry is interrupted.

## Risk register

- Provider trait churn: adding cancellation options can touch many providers. Mitigation: no-op defaults and staged adapter migration.
- Tool process leaks: hard abort can leave child processes. Mitigation: process group termination tests for bash.
- Reload regression: user cancel must not break selfdev reload handoff. Mitigation: separate `ServerReload` reason and dedicated tests.
- Duplicate Done events: cancellation races are common. Mitigation: explicit turn lifecycle state machine.
- Prompt cache regression: interrupted partial messages can alter provider payload shape. Mitigation: provider payload snapshot tests after interrupted turns.

## Done criteria

1. User interrupt, server reload, client disconnect, and Alt+B are distinct reasons.
2. Provider streams can be cooperatively cancelled where supported.
3. Tools receive cancellation and clean up child processes or locks.
4. Partial assistant text and in-flight tool calls are finalized consistently.
5. Repeated interrupts are idempotent and observable.
6. Existing soft interrupt safe-point injection still works.
7. Selfdev reload and background-tool handoff keep their current successful paths.
8. Targeted tests and release build pass.

## 원인 (Oracle high-effort 분석 + 코드 재검증)

`handle_server_event` (`src/tui/app/remote/server_events.rs`) 의 redraw
return 값은 대부분 `eager_stream_redraw && needs_redraw` 형태.

- `eager_stream_redraw = !crate::perf::tui_policy().enable_decorative_animations`
- native Linux + kitty (Full tier) → `enable_decorative_animations = true`
  → `eager_stream_redraw = false`
- WSL/minimal-tier 만 `eager_stream_redraw = true`

Server-initiated turn (auto-inject, sibling fanout) 에서는 client 가
`Request::Message` 를 보낸 적이 없어서 `begin_remote_send()` 가
호출되지 않고:

- `app.is_processing` 은 false
- `app.current_message_id` 는 None
- `app.status` 는 `Idle`

이 상태에서 첫 `TextDelta` 가 도착해도 핸들러는 status 를 안 바꾸고
(첫 분기 모두 매치 안 됨) `stream_buffer.push` 가 newline 만나기
전이면 chunk 도 안 떨어져서 `needs_redraw = false`, 결국 false 를
반환. event loop 는 redraw 안 함. 다음 idle tick (250ms ~ 1s) 만
기다리지만 그것도 `App::tick` 의 throttle 에 의해 즉시 그려지지
않을 수 있음. 사용자 입장에서는 응답이 "안 오는" 것처럼 보임.

추가로 `Done { id }` branch 는 `current_message_id == Some(id)` 가
아니면 본문 cleanup 을 skip 하므로 그 turn 의 최종 flush/commit 도
누락된다.

## Fix (이 worktree 에 적용됨)

`src/tui/app/remote/server_events.rs`:

### A) 진입부 wake-up

```rust
let is_live_stream_event = matches!(
    &event,
    ServerEvent::TextDelta { .. }
        | ServerEvent::TextReplace { .. }
        | ServerEvent::ToolStart { .. }
        | ServerEvent::ToolExec { .. }
        | ServerEvent::ToolDone { .. }
        | ServerEvent::GeneratedImage { .. }
        | ServerEvent::BatchProgress { .. }
        | ServerEvent::MessageEnd
);
let woke_server_initiated_turn = if is_live_stream_event && !app.is_processing {
    crate::logging::info(
        "M41 wake-up: server-initiated stream event arrived while client idle; \
         marking is_processing=true to force redraw",
    );
    app.is_processing = true;
    let now = Instant::now();
    app.processing_started.get_or_insert(now);
    app.last_stream_activity = Some(now);
    if matches!(app.status, ProcessingStatus::Idle) {
        app.status = ProcessingStatus::Thinking(now);
    }
    true
} else {
    false
};
```

### B) TextDelta / ToolStart / ToolExec return 값 OR

각 branch 의 return 식에 `|| woke_server_initiated_turn` 추가.
`eager_stream_redraw = false` 인 native 환경이라도 server-initiated
turn 의 첫 event 는 즉시 redraw 보장. 후속 event 는 `is_processing`
이 이미 true 이므로 woke 가 false 가 되어 평소 throttling 유지.

### C) Done 보정

```rust
let matches_woken_server_turn =
    app.current_message_id.is_none() && app.is_processing;
if app.current_message_id == Some(id) || matches_woken_server_turn {
    // 기존 flush/commit/idle cleanup
}
```

## 의문점 (다음 세션)

1. wake-up 시 `processing_started` 가 set 됐으니 `display_turn_duration_secs`
   가 0 부터 시작 — server-initiated turn 의 실제 duration 과
   misalign 가능. session 시작 시점을 server 가 별도 event 로 알리지
   않는 한 client-side 추정의 한계. 일단 wake 시점 기준으로 OK
   (사용자에게는 그 시점이 turn 시작).
2. wake-up 이후 첫 `Done { id }` 가 도착하면 `matches_woken_server_turn`
   에 잡혀 cleanup 됨. 그러나 만약 그 사이에 사용자가 새 prompt 를
   보내서 `current_message_id` 가 set 됐다면 (queue 동작), 그 Done
   은 `Some(id)` branch 로 가야 안전. 현재 분기 순서 OK 검증.
3. 장기 fix (옵션 C): protocol 에 `TurnStart { id, origin }` 추가하면
   client state 가 명시적이 되어 추정 제거. 별도 milestone 후보.

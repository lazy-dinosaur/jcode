# M22 — Subagent Same-Round Spawn Defer

**Priority**: HIGH (reopened)
**Status**: 🔴 OPEN — by-design 판정 **취소**. 사용자 항의로 재진단.

## 한 줄 요약

Main session 한 turn 안에서 subagent (`task`/`subagent` tool) 가 두 번
**진짜 동시 실행** 안 되는 현상. **fix 가능한 implementation 한계**.
M22-1 (OpenAI parallel_tool_calls toggle, default true) 는 이미 적용됨.
**남은 부분: turn loop 자체가 sequential await**.

## 2026-05-13 재진단 (1차 by-design 판정 정정)

### 1차 진단 (틀린 부분)

- "모델이 한 turn 에 task 2개 안 emit 한다" 만 보고 by-design 으로 close.
- **누락**: turn loop 자체의 sequential await 가 fix 가능한 부분이라는
  점을 무시. 모델이 multi-emit 못 하는 것도 OpenAI `parallel_tool_calls`
  토글로 일부 해결 가능 (이미 M22-1 으로 적용됨, default true).

### 진짜 구조

```
[Layer 1] Provider emit 가능 여부
  ├ Anthropic: native parallel tool_use emit 가능 (모델 결정에 따름)
  └ OpenAI Responses API: parallel_tool_calls flag
      → M22-1 으로 toggle 추가, default true (src/provider/openai.rs:712)

[Layer 2] Turn loop dispatch (src/agent/turn_streaming_broadcast.rs:702)
  └ for tool_index in 0..tool_count { ... await ... }
      → 순차. 모델이 multi emit 해도 처리는 1개씩.
      → **이게 진짜 fix point**.

[Layer 3] Task tool execute() (src/tool/task.rs)
  └ agent.run_once_capture(prompt).await
      → sub turn 완료까지 block. 이 자체는 정상.
      → 단, Layer 2 가 sequential 이므로 첫 task block = 두 번째 대기.
```

### 사용자 보고 시나리오 (가)

> "같은 round 에 두 subagent 호출 → 한쪽 silence (한참 안 시작)"

→ Layer 2 의 sequential await 가 원인. 모델이 task 2개를 한 turn 에
emit 한 경우든, 두 round 에 걸쳐 1개씩 emit 한 경우든, 사용자 UI 에서
"두 번째가 첫 번째 끝날 때까지 대기" 처럼 보임.

## fix 방향 (3 stages)

### Stage 1 — Provider emit ✅ DONE (M22-1)

- `src/config/default_file.rs:182` `openai_parallel_tool_calls = true`
- `src/provider/openai.rs:712` config 읽음
- `src/config/env_overrides.rs:427` `JCODE_OPENAI_PARALLEL_TOOL_CALLS` env

### Stage 2 — Turn loop FuturesUnordered (남음)

`src/agent/turn_streaming_broadcast.rs:702`
`src/agent/turn_streaming_mpsc.rs` (mirror)
`src/agent/turn_loops.rs` (mirror)

현재:
```rust
let tool_count = tool_calls.len();
for tool_index in 0..tool_count {
    // urgent_interrupt 체크
    let tc = &tool_calls[tool_index];
    let result = self.registry.execute(&tc.name, tc.input.clone(), ctx).await;
    // add_message_with_duration ...
}
```

목표:
```rust
let mut stream = tool_calls.iter().enumerate().map(|(i, tc)| {
    let registry = self.registry.clone();
    let ctx = ctx.for_subcall(...);
    async move {
        let result = registry.execute(&tc.name, tc.input.clone(), ctx).await;
        (i, tc.id.clone(), tc.name.clone(), result)
    }
}).collect::<FuturesUnordered<_>>();

while let Some((i, id, name, result)) = stream.next().await {
    // 결과 추가 (Anthropic tool_result 순서 보장 위해 i 기준 재정렬 가능)
}
```

### Stage 3 — Sub-agent fan-out 명시 (별도)

System prompt 에 "여러 sub-agent 가 필요하면 한 round 에 같이 호출"
권유. 이건 별개 카드 (효과 미지수).

## 위험 / 검증 포인트

| 위험 | 영향 | 완화 |
|---|---|---|
| 같은 파일 동시 write race | edit/write tool 둘이 같은 path | path 기반 mutex 또는 직렬화 옵션 |
| Anthropic tool_use/tool_result 순서 | API 가 emit 순서 보장 필요? | `i` index 로 재정렬 후 add_message |
| UI interleave | 여러 tool 진행 동시 표시 | bus event 가 이미 지원 (batch.rs 패턴) |
| urgent_interrupt 처리 | 중간 ESC 시 어떻게 cancel | join_handle abort 필요 |
| 같은 session 재진입 (subagent 자체) | recursion guard | 이미 `allow_subagent_recursion` flag |

## 사용자 검증 시나리오 (Stage 2 이후)

- `subagent` 2개 동시 호출 → 두 `Subagent starting` 로그가 ~동시각 (~1초 이내)
- 둘이 동시 진행 (bash 로 ps 확인)
- 결과가 모두 자연스럽게 main 으로 돌아옴
- ESC 한 번 → 둘 다 cancel 또는 둘 다 background 옵션

## 회귀 위험

- 같은 파일 write race (실측 + 가드 추가)
- API 호출 비용 증가 (병렬화로 throughput 늘면 cost burst)
- Logging interleave (UI 가 별로면 batch.rs UX 참고)

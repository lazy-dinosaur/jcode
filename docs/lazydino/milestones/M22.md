# M22 — Subagent Same-Round Spawn Defer

**Priority**: Medium-High → Closed (by-design)
**Status**: ✅ CLOSED, **(b) 의도된 sequential dispatch** 로 판정. fix 코드 없음.

## 한 줄 요약

Main session 한 turn 안에서 subagent (`task`/`subagent` tool) 가
두 번 즉시 spawn 안 되는 현상은 **모델 자체의 emit 패턴 + task tool 의
await-until-complete 시맨틱** 의 합성. **race/lock 버그 아님**.

## 진단 결과 (2026-05-13)

### 코드 경로 분석

- `src/agent/turn_streaming_*.rs`: `for tool_index in 0..tool_count` 로
  **sequential** dispatch (line 692). 동시 실행 가능한 비동기 fan-out
  없음.
- `src/tool/task.rs:215`: `agent.run_once_capture(...).await` 로
  subagent 완료까지 **`execute()` 가 block**. 즉 task tool 1건은 sub turn
  전체 동안 점유.
- `src/server/swarm.rs`: `swarm_members.write().await` 는 short-lived
  per-block lock. 두 동시 호출이 serialize 되지만 빠르게 완료. **lock
  holder 가 turn 종료까지 잡혀 있지 않음**.

### 실측 로그 (자연 실험)

**Case A — moose session 2026-05-13 00:28** (사용자가 두 visual
subagent 연속 spawn 시도):
```
00:28:03  Turn has 1 tool calls    Tool starting: subagent (multimodal-looker, 1.77s)
00:28:10  Turn has 1 tool calls    Tool starting: subagent (visual, 4.36s)
```
→ 한 turn 에 1개씩 emit. **모델 자체가 같은 round 에 task tool 을 둘
이상 emit 하지 않음**.

**Case B — koala session 2026-05-13 07:11~07:17** (atlas → ESC →
다음 spawn):
```
07:11:56  Subagent starting: atlas
07:14:00  Tool 'subagent' moved to background after 123.9s   ← 사용자 ESC
07:17:40  Subagent starting: quick (다음 round)              ← 정상 spawn
```
→ ESC 후 background 처리, 다음 round 에서 정상 spawn.

**전 로그 검사**: `Turn has N tool calls` 에서 N≥2 인 케이스는 빈번
(bash/read/write 등) 했으나, 그중 **task 가 2건 포함된 경우는 없음**.

### 가설 (a)/(b) 판정

- **(a) Mutex<TurnState> lock holder 의심**: **부정**. 코드/로그
  어디에도 turn 종료까지 잡히는 long-lived lock 없음.
- **(b) 의도된 single-task-per-round 정책**: **확정**. 다만 jcode 코드
  강제가 아니라 **모델 (Claude Opus 등) 의 emit 패턴** 으로 자연 발생.

## 사용자가 "deferred" 라고 느낀 이유

1. 모델이 같은 turn 에 task 2개 emit 안 함 → 첫 task 결과 보고 두 번째
   round 에서 emit.
2. Task tool 의 `execute()` 가 sub turn 끝까지 await → 사용자 UI 에서
   "첫 번째가 끝나야 두 번째가 시작" 처럼 보임.
3. ESC 로 first task → background 가도, 모델은 **next round 의 새 API
   call** 이 필요 (즉시 두 번째 spawn 안 됨, 1-2초 대기).

## Workaround (사실상 정답)

**Round 당 1 spawn**. 별도 워크어라운드가 아니라 **시스템의 자연
동작**. Docs 에 limitation ��로 명시하면 OK.

## fix 안 하는 이유

1. 진짜 race/lock 버그 없음.
2. Sequential dispatch 가 안전 (parallel tool 실행은 race condition
   위험 + Anthropic API tool_use/tool_result 순서 보장 필요).
3. 모델 emit 패턴을 jcode 가 제어하기 어려움 (system prompt 로 "한
   round 에 두 task 같이 호출하라" 가능하지만 모델이 안 따름).

## 잠재 후속 작업 (별개 카드)

같은 turn 에 정말로 N개 task 를 fan-out 으로 실행하고 싶다면:
- (A) Task tool 의 `execute()` 를 `tokio::spawn` 으로 detach + parent
  에 task_id 반환 후, parent 가 `bg wait` 로 수확.
  → swarm spawn 이 이미 이 형태. 새 모델/지시가 swarm 사용하면 가능.
- (B) Provider-side prompt 튜닝: "두 개 이상 task 가 필요하면 한 round
  에 모두 emit 하라" 권유.
  → 효과 미지수, 회귀 위험.

둘 다 **M22 와 별개 카드** 로 다루는 게 맞음 (워크어라운드로 운영해도
실제 차질 없음을 1주일+ 검증함).

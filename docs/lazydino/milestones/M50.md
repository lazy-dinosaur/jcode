# M50 — Selfdev reload post-reconnect tool readiness

**Status**: Complete (C-0/C-1/C-2/C-3/C-4 done 2026-05-18)
**Priority**: Blocks M49
**Branch stack**: `patch/m50-c*`

## Why

Server reload transport itself is expected to interrupt the active tool call and then let the client reconnect. That part is normal. The broken behavior is what can happen **after `selfdev reload` reconnects**: MCP/server-backed tools can intermittently fail to appear even when MCP configuration is valid, or the session can be marked ready before MCP tool registration has reached a final state. This destabilizes downstream workflows that depend on Figma, GitHub, Playwright, Supabase, filesystem, and other MCP-backed toolchains. M50 must be completed before returning to M49 interrupt work.

Observed risk from current code:

- `Registry::register_mcp_tools` registers only the `mcp` management tool synchronously.
- Actual server connect and `mcp__server__tool` registration happens in a background `tokio::spawn`.
- Subscribe/resume/reload flows can mark the session ready before MCP registration reaches a final state.
- `selfdev reload` intentionally interrupts the active agent-turn tool execution, waits for server ack/handoff, then relies on the resumed subscription path to rebuild session tool state.
- Manual `mcp reload` can recover some cases, which points at a lifecycle/readiness race rather than purely bad config.
- `mcp reload` unregisters existing `mcp__*` tools before reconnecting, so a failed reload can make a previously usable registry worse.

## Non-goals

- Do not redesign MCP protocol support.
- Do not treat the expected server reload interruption as a bug. The target is post-reconnect readiness and registry restoration.
- Do not change user MCP config format unless needed for retry/readiness tuning.
- Do not start M49 until M50 completion criteria are met.

## Completion criteria

1. Sessions resumed after `selfdev reload` reliably expose MCP tools after valid MCP config is present.
2. Subscribe/resume/reload readiness has an explicit MCP final state or bounded readiness barrier after reconnect.
3. Missing MCP tools can be detected and reconciled without requiring manual `mcp reload`.
4. `mcp reload` is atomic enough that reconnect failures do not unnecessarily drop still-usable tools.
5. MCP connection failures are visible in status/diagnostics, with enough detail for recovery.
6. Tests cover delayed MCP readiness, failed reload preserving old tools, and reconciliation of missing registry entries.
7. selfdev build succeeds on the final deploy tip.

### C-3. Atomic MCP reload — **DONE 2026-05-18**

Branch: `patch/m50-c3-atomic-mcp-reload`

Implemented:

- `McpManager::reload_atomic_preserving_existing` connects a candidate owned-mode manager before disconnecting existing clients.
- Explicit `mcp reload` no longer unregisters `mcp__*` tools before candidate reload success.
- When an owned-mode reload connects zero new servers and reports failures, existing manager state and registered MCP tools are preserved.
- On accepted reloads, registry replacement happens after the new manager state is ready.
- The management `mcp` tool remains registered independently of server-tool swaps.
- Shared-pool managers currently use the legacy pool reload path, because pool handles are session-keyed; C2 reconciliation still protects registry drift after reconnect.

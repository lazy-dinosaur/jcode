# M50 — Selfdev reload post-reconnect tool readiness

**Status**: Planned
**Priority**: Blocks M49
**Branch stack**: `patch/m50-c*`

## Why

Server reload transport itself is expected to interrupt the active tool call and then let the client reconnect. That part is normal. The broken behavior is what can happen **after `selfdev reload` reconnects**: MCP/server-backed tools can intermittently fail to appear even when MCP configuration is valid, or the session can be marked ready before MCP tool registration has reached a final state. This destabilizes downstream workflows that depend on Figma, GitHub, Playwright, Supabase, filesystem, and other MCP-backed toolchains. M50 must be completed before returning to M49 interrupt work.

Observed risk from current code:

- `Registry::register_mcp_tools` registers only the `mcp` management tool synchronously.
- Actual server connect and `mcp__server__tool` registration happens in a background `tokio::spawn`.
- Subscribe/resume/reload flows can mark the session ready before MCP registration reaches a final state.
- `selfdev reload` intentionally interrupts the active agent-turn tool execution, waits for server ack/handoff, then relies on the resumed subscription path to rebuild session tool state.
- Manual `mcp reload` can recover some cases, which points at a lifecycle/readiness race rather than purely bad config.
- `mcp reload` unregisters existing `mcp__*` tools before reconnecting, so a failed reload can make a previously usable registry worse.

## Non-goals

- Do not redesign MCP protocol support.
- Do not treat the expected server reload interruption as a bug. The target is post-reconnect readiness and registry restoration.
- Do not change user MCP config format unless needed for retry/readiness tuning.
- Do not start M49 until M50 completion criteria are met.

## Completion criteria

1. Sessions resumed after `selfdev reload` reliably expose MCP tools after valid MCP config is present.
2. Subscribe/resume/reload readiness has an explicit MCP final state or bounded readiness barrier after reconnect.
3. Missing MCP tools can be detected and reconciled without requiring manual `mcp reload`.
4. `mcp reload` is atomic enough that reconnect failures do not unnecessarily drop still-usable tools.
5. MCP connection failures are visible in status/diagnostics, with enough detail for recovery.
6. Tests cover delayed MCP readiness, failed reload preserving old tools, and reconciliation of missing registry entries.
7. selfdev build succeeds on the final deploy tip.

## Stages

### C-0. Reproduction fixtures and diagnostics baseline — **DONE 2026-05-18**

Branch: `patch/m50-c0-mcp-reload-fixtures`

Implemented:

- Added `Registry::mcp_registry_diagnostics()` with total tool count, management-tool presence, registered MCP server-tool count, and sorted MCP server-tool names.
- Added `register_mcp_tools_from_manager` so tests can inject a deterministic MCP manager/config without depending on global/user MCP config.
- Added a delayed stdio MCP fixture documenting the current readiness gap: the `mcp` management tool is registered synchronously, but `mcp__server__tool` entries arrive later from background registration.
- Added registry diagnostics to `mcp list` and `mcp reload` output when the management tool has access to a registry.

Validation:

- `tool::tests::mcp_registry_diagnostics_tracks_management_and_server_tools`.
- `tool::tests::register_mcp_tools_returns_before_delayed_server_tools_are_ready`.
- `tool::mcp::tests::test_list_includes_registry_diagnostics_when_registry_is_available`.
- `cargo check -p jcode`.

### C-1. MCP readiness barrier for subscribe/resume after selfdev reload — **DONE 2026-05-18**

Branch: `patch/m50-c1-mcp-readiness-barrier`

Implemented:

- `register_mcp_tools_from_manager` now waits for MCP connect/tool registration to reach a final state before returning, up to a bounded readiness timeout.
- The existing early `McpStatus` connecting event is preserved before the wait, and the final status event is still sent after tools are registered.
- If the readiness timeout expires, reconnect readiness remains bounded while the MCP registration task continues in the background.
- Added `JCODE_MCP_READINESS_TIMEOUT_MS` override with a conservative 5000ms default.
- Added `register_mcp_tools_from_manager_with_timeout` for deterministic tests without global env mutation.

Validation:

- `tool::tests::register_mcp_tools_waits_for_delayed_server_tools_within_barrier` verifies delayed MCP server tools are present when registration returns within the barrier.
- `tool::tests::register_mcp_tools_times_out_but_continues_background_registration` verifies short timeout returns promptly while background registration still completes.
- `cargo check -p jcode`.

### C-2. Registry reconciliation and auto-heal — **DONE 2026-05-18**

Branch: `patch/m50-c2-mcp-registry-reconcile`

Implemented:

- Added `Registry::reconcile_mcp_tools_from_manager`, which compares connected MCP tools from `McpManager` with actual registry entries and re-registers missing `mcp__*` tools.
- Added `McpRegistryReconcileReport` with expected, already-registered, and repaired tool counts/names.
- Added `mcp action="reconcile"` for explicit repair without a full MCP reload.
- Added auto-heal during `mcp list`, so status inspection repairs missing registry entries when connected manager state is valid.
- Added warn logs when auto-heal repairs missing tools.

Validation:

- `tool::tests::reconcile_mcp_tools_restores_missing_registry_entries` removes a registered MCP tool and verifies reconciliation restores it without reconnecting.
- `tool::mcp::tests::test_list_includes_registry_diagnostics_when_registry_is_available` continues to cover management output diagnostics after auto-heal wiring.
- `cargo check -p jcode`.

### C-3. Atomic MCP reload

Branch: `patch/m50-c3-atomic-mcp-reload`

Implement:

- Avoid unregistering old `mcp__*` tools before new tools are ready.
- Connect/build new MCP tools first, then atomically swap registry entries.
- On failure, preserve old tools when possible and report the failed new state.
- Keep `mcp` management tool always available.

Validation:

- Test failed reload preserves previously registered server tools.
- Successful reload removes stale tools only after replacement is ready.

### C-4. Retry/backoff and final UX

Branch: `patch/m50-c4-mcp-retry-status`

Implement:

- Add limited retry/backoff for transient server startup/tool-list races.
- Surface retry state and final failures in `mcp status`/diagnostics.
- Update docs/ledger with operational guidance.

Validation:

- Flaky/delayed fixture eventually registers tools without manual intervention.
- Permanent failure shows actionable status.
- Cross-crate MCP/session tests and selfdev build pass.

# M17 — Live Turn Handoff to Swarm Member

**Priority**: High (사용자 워크플로우)
**Status**: 🔴 OPEN, 미설계, 설계 결정 1회 필요

## 한 줄 요약

main session 이 긴 turn 도는 동안 사용자가 다음 메시지 보내면 queue
에 들어감. 이 queue 흐름을 swarm member 로 바로 라우팅해서 main 은
계속 진행하고 새 요청은 병렬로 처리되게 하고 싶음. (claude-code parity)

## 설계 결정 필요 (1회, ~5분)

**옵션 A: queue cancel + swarm 재발행**
- 사용자가 평소처럼 다음 메시지 enqueue
- 그 후 슬래시명령 (예: `/swarm-last`) 으로 queue 마지막 항목을
  dequeue + swarm spawn
- 장점: queue 에 한 번 들어갔다가 빼는 패턴이 직관적
- 단점: 사용자가 두 단계 (메시지 + 슬래시명령)

**옵션 B: queue skip + direct swarm**
- 슬래시명령 1줄로 queue 안 거치고 바로 swarm spawn
- **B1 (one-shot)**: `/swarm-now <prompt>` 단일 호출 → 새 swarm 즉시
  - 가장 단순 (~50줄). TUI 슬래시명령 + 기존 swarm spawn tool 호출
- **B2 (sticky toggle)**: 모드 진입하면 그 후의 모든 메시지가 swarm 으로
  - 복잡 (~200줄+). 토글 상태 관리 + UI 표시

**추천: B1 부터** (가장 단순 + 즉시 가치)

3. **main conversation 에 위임 흔적**:
   - (a) 시스템 메시지 한 줄 ("→ swarm spawned: <id>")

## 회귀 위험

- swarm spawn 이 main turn 의 ToolRegistry 를 공유 → 동시 사용 위험.
  mitigation: swarm 은 별 context 로 spawn 되므로 isolated.
- main turn 이 swarm 결과를 reference 할 수 없음 (분리된 context).
  by design, OK.

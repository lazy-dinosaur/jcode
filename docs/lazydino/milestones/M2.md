### #76 root cause coverage

issue #76 의 증상별 매핑 (현재 deploy 기준):

| 증상 | 현재 deploy 상태 | 비고 |
| --- | --- | --- |
| Subagent 3/4 무응답 | 부분 완화 | spawn retry (`24b7eaa9`) + owned_only await (`dbfc339f`) 가 일부 cover. 진짜 멈춤 원인 (transport/timeout) 은 별도 진단 필요. |
| Coordinator role error | 해결 ✅ | `e945d19c` auto-promote before cleanup. |
| 10+ 터미널 폭발 | 부분 완화 | stale cleanup (`26a5ccab`) 으로 좀비 정리 가능. spawn 횟수 자체에 대한 hard cap 은 없음. |
| 다른 cwd 에 워커 생성 | 미확인 | cwd 처리 코드 미분석. Stage 2 후보. |

즉 Stage 1 은 **이미 deployed = 행동 변화 0, 문서 정정만**. 다음
Stage 는 #76 의 남은 증상 (cwd 처리, spawn hard cap, transport
멈춤) 으로 진행해야 함.

### 다음 단계 후보 (Stage 2)

1. **cwd 처리 점검**: `swarm spawn` path 에서 worker cwd 가 어떻게
   결정되는지 추적. issue #76 의 "다른 path 에 워커 생성" 증상 검증.
2. **spawn hard cap**: 단일 coordinator 가 spawn 할 수 있는 worker
   수 상한선. 현재 무제한이면 retry path 와 결합해 폭주 가능.
3. **transport 멈춤 진단**: 워커가 ready 후 응답 없이 멈추는 케이스
   재현 → lock/event/timeout 측면 로깅 추가 (Phase C diagnostics).
4. **#135 audit 중 swarm 관련 항목 선별**.

Stage 2 는 새 worktree (또는 jcode-m2 재사용) + 새 branch 에서
